Help make the WordPress block editor work for everyone. You'll test blocks and features in the Gutenberg editor with your keyboard and a screen reader, then report what you find so the team can fix accessibility barriers.

- **Setup:** Open the [Gutenberg Playground environment](https://playground.wordpress.net/?plugin=gutenberg&url=/wp-admin/post-new.php) to start testing the latest editor in your browser — no local setup required.
- **Reference:** The [Gutenberg accessibility testing guide](https://make.wordpress.org/accessibility/gutenberg-testing/)
- **Connect:** Join [#accessibility](https://wordpress.slack.com/archives/accessibility) on Slack and introduce yourself

## Steps

1. **Pick something to test.** Choose an open issue from the [Needs Accessibility Feedback](https://github.com/WordPress/gutenberg/issues?q=is%3Aopen+is%3Aissue+label%3A%22Needs+Accessibility+Feedback%22) list, or test a block or feature yourself. The Tuesday accessibility bug scrubs in [#accessibility](https://wordpress.slack.com/archives/accessibility) are a good place to find work and learn the ropes.

2. **Test the editor.** Work through common tasks: adding and editing paragraphs, headings, images, lists, tables, and embeds, using your keyboard and a screen reader. The [Gutenberg testing guide](https://make.wordpress.org/accessibility/gutenberg-testing/) has a sample script, and the [keyboard](https://wpaccessibility.org/docs/testing/keyboard/) and [screen reader](https://wpaccessibility.org/docs/testing/screen-readers/) guides show how. You don't need to fix anything, just note what doesn't work.

3. **Search before you report.** Check the [Gutenberg issues](https://github.com/WordPress/gutenberg/issues) for your problem first. If it's already filed, add a comment confirming it still happens.

4. **Report what you found.** Comment on the issue or open a new one, including the date, the Gutenberg version, your browser, your operating system, and the assistive technology you used.

## Contribution checklist

- Tested the Gutenberg editor with your keyboard and a screen reader
- Searched existing issues before reporting
- Reported with the date, Gutenberg version, browser, operating system, and assistive technology used
- Commented on an existing issue or opened a clear new one

## What happens next

The Gutenberg team and accessibility contributors review issues on GitHub, often during the Tuesday bug scrubs, and decide what needs fixing. You don't need to write code or fix the issue yourself. If your report gets no response, raise it in [#accessibility](https://wordpress.slack.com/archives/accessibility) or bring it to a bug scrub.

Once you've reported one issue, pick up another from the Needs Accessibility Feedback list, or join a bug scrub to test alongside the team.
